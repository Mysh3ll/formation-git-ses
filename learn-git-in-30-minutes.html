I. Les bases

Git est une collection d'utilitaires de ligne de commande qui suivent et enregistrent les modifications dans les fichiers (le plus souvent le code source, mais vous pouvez suivre tout ce que vous souhaitez). Avec lui, vous pouvez restaurer les anciennes versions de votre projet, comparer, analyser, fusionner les modifications et plus encore. Ce processus est appelé contrôle de version. Il existe un certain nombre de systèmes de contrôle de version qui font ce travail. Vous avez peut-être entendu certains d'entre eux - SVN, Mercurial, Perforce, CVS, Bitkeeper et plus encore.

Git est décentralisé, ce qui signifie qu'il ne dépend pas d'un serveur central pour conserver les anciennes versions de vos fichiers. Au lieu de cela, il fonctionne entièrement localement en stockant ces données sous forme de dossier sur votre disque dur, que nous appelons un dépôt. Cependant, vous pouvez stocker une copie de votre dépôt en ligne, ce qui permet à plusieurs personnes de collaborer et de travailler facilement sur le même code. C'est à cela que servent des sites Web comme GitHub et BitBucket.

1. Installer Git

L'installation de Git sur votre machine est simple:
  • ➔ Linux - Ouvrez simplement un nouveau terminal et installez git via le gestionnaire de paquets de votre distribution. Pour Ubuntu, la commande est: sudo apt-get install git
  • ➔ Windows - nous recommandons git pour windows car il offre à la fois un client GUI et un émulateur de ligne de commande BASH.
  • ➔ OS X - La façon la plus simple est d'installer homebrew, puis d'exécuter brew install git depuis votre terminal.

Si vous êtes un débutant absolu, un client graphique git est un must. Nous recommandons TortoiseGit, mais il existe de nombreux autres bons et gratuits en ligne. Connaître les commandes de base de git est toujours important même si vous utilisez une application GUI, donc pour le reste de cette leçon, ce sera notre seul objectif.

2. Configurer Git

Maintenant que nous avons installé git sur notre ordinateur, nous devrons ajouter quelques configurations rapides. Il existe de nombreuses options qui peuvent être manipulées, mais nous allons configurer les plus importantes: notre nom d'utilisateur et notre adresse e-mail. Ouvrez un terminal et exécutez ces commandes: 

					
						# Configuration du nom d'utilisateur
						git config --global user.name "My Name"

						# Configuration de l'adresse e-mail
						git config --global user.email myEmail@example.com

Chaque action que nous faisons dans Git aura désormais un tampon avec notre nom et notre adresse. De cette façon, les utilisateurs savent toujours qui a fait quoi et tout est beaucoup plus organisé.

3. Créer un nouveau dépôt

Comme nous l'avons mentionné précédemment, git stocke ses fichiers et son historique directement sous forme de dossier dans votre projet. Pour configurer un nouveau référentiel, nous devons ouvrir un terminal, accéder à notre répertoire de projet et exécuter: git init


Cela activera Git pour ce dossier particulier et créera un répertoire .git caché où l'historique et la configuration du dépôt seront stockés.

Créez un dossier sur votre bureau appelé git_exercice, ouvrez un nouveau terminal et entrez les informations suivantes: 

					
						# Ouverture du dossier
						cd Desktop/git_exercice/

						# Création du nouveau dépôt
						git init

La ligne de commande doit répondre par quelque chose comme: 

					
						Initialized empty Git repository in /home/user/Desktop/git_exercise/.git/

Cela signifie que notre dépôt a été créé avec succès mais qu'il est toujours vide. Créez maintenant un fichier texte appelé hello.txt et enregistrez-le dans le dossier git_exercice.

4. Vérifier le status

Git status est une autre commande incontournable qui renvoie des informations sur l'état actuel du dépôt: tout est à jour, ce qui est nouveau, ce qui a changé, etc. L'exécution de git status dans notre dépôt nouvellement créé devrait renvoyer ce qui suit: 

					
						git status

						On branch master

						Initial commit

						Untracked files:
						  (use "git add ..." to include in what will be committed)

							hello.txt

Le message renvoyé indique que hello.txt n'est pas suivi. Cela signifie que le fichier est nouveau et Git ne sait pas encore s'il doit garder une trace des modifications apportées à ce fichier ou simplement l'ignorer. Pour accuser réception du nouveau fichier, nous devons le mettre dans l'Index (stage).

5. Staging

Git a le concept de "zone de transit". Vous pouvez penser à cela comme un sac vide, qui contient les modifications que vous souhaitez valider. Il commence vide, mais vous pouvez y ajouter des fichiers (ou même des lignes simples et des parties de fichiers) avec la commande git add, et enfin tout valider (créer un instantané) avec git commit.

Dans notre cas, nous n'avons qu'un seul fichier alors ajoutons le:

git add hello.txt

Si nous voulons tout ajouter dans le répertoire, nous pouvons utiliser:

git add -A

Une nouvelle vérification de l'état devrait renvoyer une réponse différente de celle d'avant. 

					
						git status

						On branch master

						Initial commit

						Changes to be committed:
						  (use "git rm --cached ..." to unstage)

							new file:   hello.txt

Notre fichier est prêt à être "commité". Le message d'état nous indique également ce qui a changé sur les fichiers dans la zone de transfert - dans ce cas, son nouveau fichier, mais il peut être modifié ou supprimé, selon ce qui est arrivé à un fichier depuis le dernier git add.

6. Faire un commit

Un commit représente l'état de notre dépôt à un moment donné. C'est comme un instantané, auquel nous pouvons revenir pour voir comment c'était lorsque nous l'avons pris.

Pour créer un nouveau commit, nous devons ajouter au moins une modification à la zone de transfert (nous venons de le faire avec git add) et exécuter ce qui suit:

git commit -m "Initial commit."

Cela va créer un nouveau commit avec toutes les modifications de la zone de transfert (en ajoutant hello.txt). La partie -m "Initial commmit" est une description écrite par l'utilisateur qui résume les modifications apportées dans cette validation. Il est considéré comme une bonne pratique de valider souvent et d'écrire toujours des messages de validation significatifs.

II. Les dépôts distants

En ce moment, notre commit est local - il n'existe que dans le dossier .git. Bien qu'un dépôt local soit utile en soi, dans la plupart des cas, nous voudrons partager notre travail et le déployer sur un serveur ou un service d'hébergement de dépôts.

1. Se connecter à un dépôt

Afin de télécharger quelque chose sur un dépôt distant, nous devons d'abord établir une connexion avec lui. Pour les besoins de ce didacticiel, l'adresse de notre référentiel sera https://github.com/Mysh3ll/formation-git-ses. Nous vous conseillons d'aller de l'avant et de créer votre propre dépôt vide sur GitHub, BitBucket ou tout autre service. L'enregistrement et la configuration peuvent prendre un certain temps, mais tous les services proposent de bons guides pas à pas pour vous aider.

Pour lier notre dépôt local à celui sur GitHub, nous exécutons la ligne suivante dans le terminal: 

					
						# Ce n'est qu'un exemple. Remplacez l'URL par votre propre adresse de dépôt.
						git remote add origin https://github.com/Mysh3ll/formation-git-ses.git

Un projet peut avoir plusieurs dépôts distants en même temps. Pour pouvoir les différencier, nous leur donnons des noms différents. Traditionnellement, le dépôt distant principal dans git est appelé origin.

2. Envoyer son travail vers un dépôt

Il est maintenant temps de transférer nos validations locales sur le serveur. Ce processus est appelé push, et est effectué chaque fois que nous voulons mettre à jour le dépôt distant.

La commande Git pour ce faire est git push et prend deux paramètres - le nom du dépôt distant (nous avons appelé la nôtre origin) et la branche vers laquelle pousser (master est la branche par défaut pour chaque dépôt). 

					
						git push origin master

						Counting objects: 3, done.
						Writing objects: 100% (3/3), 212 bytes | 0 bytes/s, done.
						Total 3 (delta 0), reused 0 (delta 0)
						To https://github.com/Mysh3ll/formation-git-ses.git
						* [new branch]      master -> master

Selon le service que vous utilisez, vous devrez vous authentifier pour que le push fonctionne. Si tout a été fait correctement, lorsque vous allez dans votre navigateur Web dans le dépôt distant créé précédemment, hello.txt devrait y être disponible.

3. Cloner un dépôt

À ce stade, les gens peuvent voir et parcourir votre dépôt distant sur Github. Ils peuvent le télécharger localement et avoir une copie entièrement fonctionnelle de votre projet avec la commande git clone:

git clone https://github.com/Mysh3ll/formation-git-ses.git

Un nouveau dépôt local est automatiquement créé, avec la version de github configurée comme un dépôt distant.

4. Obtenir les modifications à partir d'un dépôt

Si vous apportez des mises à jour à votre dépôt, les utilisateurs peuvent télécharger vos modifications avec une seule commande - pull: 

					
						$ git pull origin master

						From https://github.com/Mysh3ll/formation-git-ses
						 * branch            master     -> FETCH_HEAD
						Already up-to-date.

Étant donné que personne d'autre n'a "commité" depuis le clonage, il n'y a eu aucun changement à télécharger.

III. Les branches

Lors du développement d'une nouvelle fonctionnalité, il est considéré comme une bonne pratique de travailler sur une copie du projet d'origine, appelée branche. Les branches ont leur propre historique et isolent leurs modifications les unes des autres, jusqu'à ce que vous décidiez de les fusionner à nouveau. Cela se fait pour deux raisons:

  • ➔ Une version stable et déjà fonctionnelle du code ne sera pas rompue.
  • ➔ De nombreuses fonctionnalités peuvent être développées en toute sécurité simultanément par différentes personnes.
  • ➔ Les développeurs peuvent travailler sur leur propre branche, sans risque de modification de leur base de code en raison du travail de quelqu'un d'autre.
  • ➔ En cas de doute sur ce qui est le mieux, plusieurs versions de la même fonctionnalité peuvent être développées sur des branches distinctes, puis comparées.

1. Créer une nouvelle branche

La branche par défaut de chaque dépôt est appelée master. Pour créer des branches supplémentaires, utilisez la commande git branch "maBranche" :

git branch amazing_new_feature

Cela crée simplement la nouvelle branche, qui à ce stade est exactement la même que notre master.

2. Changer de branche

Maintenant, lorsque nous exécutons git branch, nous verrons qu'il y a deux options disponibles: 

					
                        git branch
                          amazing_new_feature
                        * master

Master est la branche actuelle et est marquée d'un astérisque. Cependant, nous voulons travailler sur nos nouvelles fonctionnalités incroyables, nous devons donc passer à l'autre branche. Cela se fait avec la commande git checkout, en attendant un paramètre - la branche vers laquelle basculer.

git checkout amazing_new_feature

3. Fusionner une branche

Notre "nouvelle fonctionnalité incroyable" ne sera qu'un autre fichier texte appelé feature.txt. Nous allons le créer, l'ajouter et le "commiter". 

					
                        # Ajout du nouveau fichier
                        git add feature.txt

                        # Commit de la nouvelle fonctionnalité
                        git commit -m "Nouvelle fonctionnalité terminée."

La nouvelle fonctionnalité est terminée, nous pouvons revenir à la branche master.

git checkout master

Maintenant, si nous ouvrons notre projet dans le navigateur de fichiers, nous remarquerons que feature.txt a disparu. C'est parce que nous sommes de retour dans la branche master, et ici feature.txt n'a jamais été créé. Pour l'inclure, nous devons fusionner les deux branches ensemble avec la commande git merge, en appliquant les modifications apportées dans amazing_new_feature à la version principale du projet.

git merge amazing_new_feature

La branche master est maintenant à jour. La branche awesome_new_feature n'est plus nécessaire et peut être supprimée.

git branch -d amazing_new_feature
